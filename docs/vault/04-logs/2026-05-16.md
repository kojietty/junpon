## 2026-05-16 18:00

### 目的
2キーモード (D/J) の追加とランキング分離

### 決定 / 理由 / 代替案

**決定**: 4キー (S/D/J/K) と並行で 2キー (D/J) モードを追加。ハイスコア・サーバランキング共にモード別に完全分離する。

**理由**:
- 4キーは反射神経への要求が高くカジュアル層が落ちやすい。間口を広げるための簡単モードが欲しかった
- 異なるキー数のスコアを混ぜると比較不能になるためランキング分離は必須

**代替案として却下**:
- 独立 `ModeSelectScene` を挟む案 → タイトル → モード → ゲームと 1 段増えるのに対し情報量が乏しいため不採用
- `scores_2key` テーブル新設 → スキーマと worker のコードパスが二重化する。`mode` カラム追加なら GET の WHERE 1 行と INSERT 1 列の追加で済む

### 実施内容

#### 設定 / 型
- [[../02-tech/architecture|architecture]] の `LANES` / `DIFFICULTY.laneCount` を撤去し `MODES: Record<GameMode, {count, keys, colorIdx}>` ルックアップに統一
- `GameMode = 2 | 4`、`DEFAULT_MODE = 4`、`parseMode(unknown)` ヘルパ
- `STORAGE_KEYS.highScore` を `(mode) => 'jungle-tap:highscore:Nkey'` の関数化
- `STORAGE_KEYS.lastMode` を追加（タイトル → RANKING の初期タブやキーボードショートカット用）

#### シーン間データフロー
- TitleScene → GameScene: `{ mode }`
- GameScene → GameOverScene: `{ score, mode }`
- GameOverScene → GameScene (RETRY): `{ mode }`
- GameScene 一時停止 → やりなおす: `{ mode }`
- GameOverScene → RankingScene: `{ myScore, myNickname, initialMode }`
- TitleScene → RankingScene: `{ initialMode: lastMode }`

#### TitleScene
- START を **2KEY START / 4KEY START** の 2 つに分割（pink / blue）。クリック時に `lastMode` を localStorage に保存
- BEST 表示を `2KEY ###   4KEY ###` の一行並記に変更
- ENTER/SPACE は `lastMode` で開始

#### RankingScene
- `2KEY / 4KEY` ピル型タブを y=150 に配置（active = モード色塗り、inactive = 透過 + outline）
- 初回はアクティブタブのみフェッチ、切替時に他方をフェッチして `entriesCache: Record<GameMode, RankingEntry[]>` にキャッシュ
- 行コンテナを `rowContainers[]` で追跡し、タブ切替時に destroy → 再描画

#### バックエンド (worker.ts) + migration
- `migrations/0001_add_mode_column.sql`: `ALTER TABLE scores ADD COLUMN mode INTEGER NOT NULL DEFAULT 4;` + `idx_scores_mode_score`
- GET: `?mode=2|4` を必須、WHERE 句追加。デフォルト 4（旧クライアント互換）
- POST: body の `mode` を 2|4 で必須検証、INSERT に追加
- レート制限は **デバイス単位グローバル** を維持（30 秒 / モード横断、スパム予算が倍にならないように）

#### BootScene
- 旧 `'jungle-tap:highscore'` → `'jungle-tap:highscore:4key'` への冪等な移行関数を追加

### 技術メモ
- `Phaser.Scene.cache` という built-in プロパティがあるため、`RankingScene` 側のキャッシュフィールドは `entriesCache` に改名する必要があった
- `MODES[2].keys` は長さ 2 の readonly tuple なので `keys[lane]` が `string | undefined` になる。Character は spawner が valid な lane しか出さない前提で `as string` キャストで通している
- `LaneKey = 0|1|2|3` は据え置き。2キーモードは 0|1 サブセットのみ使用するためキャストは安全

## 2026-05-16 21:00

### 目的
最大 10 人ルームコード対戦モード (準同期 VS) の追加

### 決定 / 理由 / 代替案

**決定**: ロビーで集合 → ホストが START → 同シードで全員 60 秒プレイ → 結果でスコア突き合わせの **準同期方式** を採用。Cloudflare 無料枠厳守のため Durable Objects / 常時 WebSocket は不採用。

**理由**:
- Durable Objects は無料枠外 ($5/月)。個人開発で課金を増やしたくない
- 準同期ならプレイ中は通信ゼロ。Workers + D1 のステートレス API + クライアントポーリングで 1 セッション ≈ 160 req、無料枠 100k req/日で 625 ルーム/日まで耐える
- ノーツ系列を mulberry32 (32bit シード) で決定論化することで、通信なしに全端末で同じ盤面が再現できる

**代替案として却下**:
- 完全リアルタイム (Durable Objects + WebSocket) → 月額課金が発生
- ローカル対戦 (1 端末で 2 人キー分割) → 大人数化が不可能、フレンド対戦の体験にならない
- 非同期チャレンジ (シード共有 URL を投げ合う) → 同時プレイの緊張感がない

### 実施内容

#### スキーマ (migrations/0002_create_rooms.sql)
- `rooms` テーブル: code, mode, seed, max_players (2-10), host_token, host_device, start_at, created_at, expires_at
- `room_participants` テーブル: code, slot, token, device, nickname, score, plays, joined_at
- `idx_rooms_expires` / `idx_participants_code` / `idx_participants_token`
- TTL: 10 分。各 API ハンドラ先頭で確率 1/20 で lazy GC

#### Worker API (worker.ts)
- `POST /api/rooms` ルーム作成 (mode/maxPlayers/nickname)、device 10 秒 RL、コード衝突は最大 5 回リトライ
- `POST /api/rooms/:code/join` 参加、device 3 秒 RL、満員 / 重複 / 開始済みを 409
- `POST /api/rooms/:code/start` host のみ。`start_at = now + 5s`、参加 < 2 で 400
- `GET /api/rooms/:code` ポーリング用。status を動的計算 (waiting/starting/playing/finished/expired)。`start_at + 120s` 経過後の未送信は dnf=true / score=0 で集計
- `POST /api/rooms/:code/score` token から participant 特定。score 上限 500、plays 上限 600、受付窓 `start_at+55s ~ +120s`

#### 決定論ノーツ ([[../02-tech/architecture]] に反映済)
- `src/systems/SeededRng.ts` に mulberry32 を実装
- `CharacterStack` に `rng?: () => number` オプションを追加 (省略時 Math.random フォールバック)。差分は spawnAtTop の 1 行のみ
- `tickRefill` は時刻ベースで rng を呼ばないため、フレームレート差・操作タイミング差に依存しない

#### クライアント新規シーン
- `VsMenuScene` CREATE / JOIN 二択。CREATE は mode/maxPlayers/nickname、JOIN はコード/nickname
- `VsLobbyScene` host: コード 6 桁を 3-3 区切りで大表示、参加者リスト、人数 ≥ 2 で START 有効、TTL カウントダウン / guest: 同リスト + ホスト待ち
- `VsCountdownScene` サーバ `startAt - serverOffset` 到達まで sleep → 3-2-1-GO → GameScene へ
- `VsResultScene` 結果ポーリング、順位表 (同点同順位)、自分ハイライト、WIN / DRAW / N位 ラベル

#### GameScene 差分
- `vs?: VsContext` を受け取り、`mulberry32(vs.seed)` を CharacterStack へ注入
- vs 時は `startCountdown` をスキップして即 `startGame()` (カウントダウンは VsCountdownScene 側)
- `onInput` 冒頭で `this.plays++` (送信時の不正対策値)
- `endSession` で vs ありなら VsResultScene へ遷移
- ポーズメニューを vs 時は「リタイア」「対戦から離脱」に置換 (score=0 で確定)

#### TitleScene
- "VS PLAY (最大10人)" ボタンを 4 番目に追加。レイアウトを 1 段繰り上げ

#### サービス
- `src/services/VsService.ts`: createRoom / joinRoom / getRoom / startRoom / submitScore + `VsApiError`
- `formatRoomCode` (3-3 区切り) と `sanitizeRoomCode` ヘルパ

### 技術メモ
- Phaser.Scene には built-in の `data` プロパティ (DataManager) があるため、シーン data を private フィールドに保存する場合は `params` などへリネームが必要 (既存 RankingScene の `cache` → `entriesCache` と同パターン)
- ルームコード文字種は `ABCDEFGHJKMNPQRSTUVWXYZ23456789` (28文字、`0OIL1` を除外)。28^6 ≈ 4.8 億で同時 1000 ルームでも衝突確率 0.0002%
- 時刻同期: 各レスポンスに `serverNow` を含め、クライアントは `offset = serverNow - localNow` を算出。ノーツ決定論なのでクロックずれ ±500ms は許容範囲
- D1 の FK CASCADE はデフォルト off。lazy GC では `room_participants` と `rooms` を別 statement で `batch()` 削除する設計に
